This document tracks the specific errors encountered during the 10-node deployment and the solutions implemented.

---

### Issue 1: Worker Nodes Unable to Join Cluster
*   **Symptom**: Agent service fails on workers; `kubectl get nodes` only shows the Control Planes.
*   **Root Cause 1 (Networking)**: GCP default firewall blocks port `6443` (K8s API) and `2379-2380` (etcd).
*   **Root Cause 2 (NTP)**: Clock skew between nodes caused TLS certificate validation to fail.
*   **The Fix**:
    1.  **GCP Firewall**: Created a VPC rule to allow internal traffic on ports `6443`, `10250` (Kubelet), and `2379:2380`.
    2.  **Time Sync**: Forced NTP sync on all 10 nodes:
        ```bash
        sudo timedatectl set-ntp yes
        ```

---

### Issue 2: Docker Rejecting Harbor (Insecure Registry)
*   **Symptom**: GitLab Runner CI job fails with: `server gave HTTP response to HTTPS client` when pushing images.
*   **Root Cause**: Docker requires HTTPS by default. Harbor was running on an internal IP without a trusted CA certificate.
*   **The Fix (Runner Node)**:
    Added the Harbor IP to the Docker daemon configuration `/etc/docker/daemon.json`:
    ```json
    {
      "insecure-registries" : ["<HARBOR_IP>:30002"]
    }
    ```
    Then restart the service:
    `sudo systemctl restart docker`.

---

### Issue 3: K3s Pods "ImagePullBackOff"
*   **Symptom**: Pods fail to pull images from Harbor, even though the Runner successfully pushed them.
*   **Root Cause**: K3s uses `containerd`, which ignores Docker’s `daemon.json`.
*   **The Fix (All K3s Nodes)**:
    Edited `/etc/rancher/k3s/registries.yaml` on all 6 cluster nodes:
    ```yaml
    mirrors:
      "<HARBOR_IP>:30002":
        endpoint:
          - "http://<HARBOR_IP>:30002"
    configs:
      "<HARBOR_IP>:30002":
        tls:
          insecure_skip_verify: true
    ```
    Then restart the service:
    `sudo systemctl restart k3s`.

---

### Issue 4: GitLab Runner Docker-in-Docker (DinD) Permission
*   **Symptom**: `permission denied` when trying to run docker commands inside the CI pipeline.
*   **The Fix**:
    ```bash
    sudo usermod -aG docker gitlab-runner
    sudo chmod 666 /var/run/docker.sock
    ```

---

### Issue 5: SonarQube ElasticSearch Memory Limits
*   **Symptom**: SonarQube container crashes immediately on start.
*   **Root Cause**: ElasticSearch (inside SonarQube) requires a higher `max_map_count` than the Linux default.
*   **The Fix**:
    Run on the SonarQube node:
    ```bash
    sudo sysctl -w vm.max_map_count=262144
    ```
